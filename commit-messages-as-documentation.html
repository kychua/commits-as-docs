Commit messages as documentation

  • summarise changes

  • provide context: when, who, why

  • don’t become outdated

  • searchable

Let’s say one day, you used git blame and discovered the commit that caused the bug…​

…​and discovered the commit was 1000 lines long, including 3 new features

…​not that useful, is it?

Atomic

Only one logical change

  • whitespace fixes, refactorings separate from functional changes

  • different features

Atomic commits are easier to…​

  • review, debug, revert

Coherent

Commits should not break build

  • test updates in the same commit

  • code works as is

Coherent commits

  • allow you to use git blame, git bisect etc.

Commit messages

  • summary of code changes

  • provide context behind the code change

  • make sense without seeing code diffs & external resources

  • consistent, correct spelling

  • correct formatting for readability

  • make clear which part of code is affected
